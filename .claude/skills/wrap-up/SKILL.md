# Wrap-Up

セッション終了時のドキュメント整備パイプラインを実行する。以下のステップを順番に行うこと。

## Step 1: 変更の把握

```bash
git status
git diff HEAD
```

未コミット・未ステージの変更を把握し、ファイル群を**論理的な単位**に分類する。
1ファイルが複数の目的を持つ場合はその旨を記録しておく。

## Step 2: Conventional Commit でコミット

変更を以下の型でグループ化し、グループごとにコミットする：

| 型         | 用途                                             |
| ---------- | ------------------------------------------------ |
| `feat`     | 新機能                                           |
| `fix`      | バグ修正                                         |
| `docs`     | ドキュメントのみの変更                           |
| `style`    | フォーマット・スタイルの修正（ロジック変更なし） |
| `refactor` | リファクタリング                                 |
| `test`     | テストの追加・修正                               |
| `chore`    | ビルド・設定・ツール変更                         |

コミットメッセージ形式：

```
<type>(<scope>): <subject>
```

例：`feat(hero): add logo random swap animation`

コミット手順：

```bash
git add <対象ファイル>
git commit -m "<conventional commit message>"
```

`git commit` はユーザーの確認後に実行すること。自動的にコミットしない。

## Step 3: JSDoc/TSDoc の更新

Step 1 で変更されたファイルの中から、公開関数・型・クラスを持つ `.ts` / `.tsx` ファイルを特定する。
各ファイルを Read で確認し、以下の条件を満たす場合にのみ JSDoc/TSDoc を追加・更新する：

- `@param` / `@returns` が実態と乖離している
- 公開 API に JSDoc が存在しない
- 型定義が変わった

スタイルルール：

- 1行の説明で十分な場合は複数行にしない
- 「何をするか」ではなく「なぜ・どう使うか」を書く
- コメントがないほうが明快なら書かない（CLAUDE.md のコメントルール準拠）

## Step 4: セッションサマリーの生成

`docs/sessions/YYYY-MM-DD.md` を生成する（YYYY-MM-DD は今日の日付）。

ファイルが既に存在する場合は末尾に追記する。

内容テンプレート：

```markdown
# Session YYYY-MM-DD

## 実施内容

- （変更の概要を箇条書き）

## コミット

| ハッシュ | メッセージ   |
| -------- | ------------ |
| abc1234  | feat(x): ... |

## 残課題

- （未解決・次回に持ち越す事項）

## 備考

- （判断の背景・注意点など）
```

コミットハッシュは `git log --oneline -10` で取得する。

## 注意事項

- `git push` はこのコマンドでは実行しない。ユーザーが判断する。
- JSDoc の更新は変更ファイルに限定し、関係のないファイルに触れない。
- doc-curator サブエージェントが利用可能な場合、Step 3〜5 をそちらに委譲する。
